# 答题纸素材 03（对应 2.1 共享内存线程安全 5 分）

阶段 03 把服务器的用户表从 `chatserver.c` 里的普通全局数组迁入 **POSIX 共享内存**。
原因有二：一是为阶段 04「主线程 select 分派 + 各类请求各起工作线程」做准备——多个
执行流要看到同一份用户表，必须放进可共享、带锁保护的内存区；二是共享内存
（`shm_open`+`mmap`）相比管道是真正的零拷贝共享，读写直接落在同一物理页上。

创建过程（`src/user_store.c`）：`shm_open("/chatroom_lwj_users", O_CREAT|O_RDWR, 0600)`
拿到一个 fd；`fchmod(fd,0600)` 把已存在对象的权限收紧（个别平台对 shm fd 的 fchmod
不支持，则退而 `fstat` 确认未对组/其他开放）；`ftruncate(fd, sizeof(ChatUserStore))`
把对象定长到结构体大小；`mmap(PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0)` 映射进地址空间。
对象对应 Linux 下的 `/dev/shm/chatroom_lwj_users`。

线程安全靠**放在共享区内**的一把 `pthread_mutex_t`。普通互斥锁默认只在本进程内有效，
要让它跨进程，必须先 `pthread_mutexattr_setpshared(&attr, PTHREAD_PROCESS_SHARED)`
再 `pthread_mutex_init`，否则别的进程/线程拿到的锁状态无意义。这把锁保护
`user_count` 与整张 `users[]`：注册查重+插入、登录改 online/fifo、登出置 online=0、
聊天投递前取一致快照，全部在临界区内完成；FIFO 回包这类阻塞操作放到锁外，缩小锁粒度。

验证：`/dev/shm/chatroom_lwj_users` 存在且权限 `600`；`server.log` 出现
`shm user store ready ...` 与 `process-shared` 初始化记录；用同名用户连续注册两次，
第二次返回 `username already exists`，证明用户表确实落在跨客户端持久的共享区里而非
单次进程内存；登录/登出仍各回 `login ok`/`logout ok`；服务器正常退出后共享对象被
`shm_unlink` 删除。本阶段**尚不能**证明真正的多线程并发正确性——服务器仍是单线程
select，并发压力测试要到阶段 04 工作线程引入后才有意义；这里的加锁是为那时预留正确性。
